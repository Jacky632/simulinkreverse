# Simulink C Reconstruction Helper

This project reads C and header files generated by Simulink Coder and writes a
MATLAB script that builds an approximate Simulink model with API calls such as
`new_system`, `add_block`, `set_param`, `add_line`, and `save_system`.

## Usage

```bash
python main.py --input ccode --output generated_model_builder.m --model-name reconstructed_model

python main.py --input ccode --output generated_model_builder.m --model-name reconstructed_model --verbose > reconstruction.log 2>&1
```

Optional flags:

```bash
python main.py --input ccode --output generated_model_builder.m --model-name reconstructed_model --verbose
python main.py --input ccode --output generated_model_builder.m --model-name reconstructed_model --debug
python main.py --input ccode --output generated_model_builder.m --model-name reconstructed_model --fallback-matlab-functions
python main.py --input ccode --output generated_model_builder.m --generate-layout-variants
```

The layout-variant command writes seven independent MATLAB scripts:

- `generated_model_builder_layout1_hierarchical.m`
- `generated_model_builder_layout2_left_to_right_signal_flow.m`
- `generated_model_builder_layout3_controller_plant_grouped.m`
- `generated_model_builder_layout4_grid_aligned.m`
- `generated_model_builder_layout5_subsystem_modular.m`
- `generated_model_builder_layout6_simple_subsystems.m`
- `generated_model_builder_layout7_complex_subsystems.m`

Each variant saves a separate `.slx` model, prints generation progress in
MATLAB, validates generated block and signal-line counts, and includes the word
`inferred` in inferred helper block names. Layouts 6 and 7 prefer actual
subsystems; layout 6 uses a few coarse subsystems, while layout 7 uses a more
granular subsystem decomposition.

Run the generated script in MATLAB with Simulink installed:

```matlab
run('generated_model_builder.m')
```

It creates and saves `reconstructed_model.slx`.

## What It Does

- Recursively scans the input folder for `.c` and `.h` files.
- Ignores common binary files, build artifacts, generated reports, and temporary
  files.
- Extracts function definitions, step/initialize/terminate functions, structs,
  parameters, assignments, function calls, state updates, and Simulink Coder
  trace comments.
- Builds a MATLAB-independent intermediate representation with blocks, signals,
  parameters, states, connections, annotations, and warnings.
- Maps recognizable expressions to Simulink blocks:
  - `y = u` to pass-through/direct-style signal conversion
  - `y = K * u` or `y = u * K` to Gain
  - `y = a + b` and `y = a - b` to Sum
  - `y = a * b` to Product
  - `y = 5.0` or `y = P.Constant_Value` to Constant
  - generated Integrator and derivative patterns to Integrator
  - generated saturation `if/else` patterns to Saturation
  - generated Unit Delay state patterns to Unit Delay
  - unknown code to annotations or MATLAB Function fallback blocks
- Places Inports on the left, computation in the middle, and Outports on the
  right using dependency depth.

## Important Limitations

Simulink Coder output is not a lossless representation of the original `.slx`.
The generated model is an approximate visualization and a starting point for
human review, not a guaranteed identical reconstruction.

Information that may be lost or changed includes exact block names, hierarchy,
layout, masks, sample times, solver settings, data types, annotations, variant
conditions, bus definitions, lookup table metadata, and library links. Some C
logic cannot be uniquely mapped back to Simulink blocks. The tool logs inferred
facts separately from certain facts recovered from structs and Simulink Coder
trace comments.

## Tests

```bash
python -m pytest
```

The tests include a small sample C model under `tests/sample_ccode/` and cover
file loading, function detection, assignment detection, expression mapping,
MATLAB API command generation, and the full pipeline.
